declarativeNetRequest — URL Timing Oracle

PROOF OF CONCEPT

Vulnerability Summary

A Chrome extension holding only the declarativeNetRequest and storage permissions can silently leak the full URL of any open tab by exploiting the measurable timing difference between a blocked request (net::ERR_BLOCKED_BY_CLIENT, ~10–30 ms) and an allowed request (~50–100 ms+). A binary search over the character set resolves each character in O(log n) iterations.

Affected versions: Chrome 144.0.7559.97 (Stable) through 146.0.7653.0 (Canary) on Windows 11 24H2. Regression introduced in the commit enabling regex rule evaluation for dynamic rules.

Extension Permissions Used

declarativeNetRequest storage

No tabs, no host_permissions, no user interaction required.

How It Works

  1. Extension sets a dynamic DNR blocking rule with a regex pattern targeting a character subset at position i of the URL.
  2. Calls chrome.tabs.reload and starts a high-resolution timer.
  3. Listens for chrome.tabs.onUpdated status === "complete".
  4. If elapsed time < threshold → URL matches → character is in the left half of the search set.
  5. Repeat binary search until the character is resolved, then advance to position i+1.
  6. Leaked URL is saved to chrome.storage.local and zipped + POSTed to the collection server.

Reproduction Steps

  1. Start the collection server: cd server && npm i && npm start
  2. Open a target URL (e.g. https://accounts.google.com/callback?code=secret) in a tab and move it to position 1.
  3. Load the unpacked extension from this directory in Chrome (chrome://extensions → Developer mode → Load unpacked).
  4. Open the extension's service worker DevTools and watch the console.
  5. Visit the collection dashboard to see the exfiltrated URL.

Impact

OAuth tokens · session tokens · password-reset links · private share URLs · sensitive search queries — any secret embedded in a URL is silently exfiltrated with zero user interaction and no visible UI indicator.
